# 无理由退货字段映射问题 - 诊断与修复

## 问题描述

导入时填写"7天无理由退货（一次性包装破损不支持）"，但导出时显示的是"1300030895"（对应"不支持7天无理由退货"），而正确的应该是"1300030902"。

## 完整映射关系

```
1300030895 - 不支持7天无理由退货
1300030902 - 7天无理由退货（一次性包装破损不支持）
1300030903 - 7天无理由退货（激活后不支持）
1300030904 - 7天无理由退货（使用后不支持）
1300030905 - 7天无理由退货（安装后不支持）
1300030906 - 7天无理由退货（定制类不支持）
1300030901 - 7天无理由退货（通用）
```

## 可能的原因

### 原因1：数据库中旧数据的问题

如果你的数据库中已经有旧数据，可能存在以下问题：

1. **no_reason_return_tag_id 字段为 NULL**
   - 旧数据可能没有这个字段
   - 数据库迁移时没有正确更新

2. **no_reason_return 和 no_reason_return_tag_id 不一致**
   - 旧逻辑只使用布尔值，新的标签ID可能没有正确同步

### 原因2：Excel文件中的文本问题

1. **包含不可见字符**
   - 从其他系统复制粘贴的数据可能包含特殊字符
   - BOM标记、零宽字符等

2. **全角/半角括号混用**
   - 中文全角括号：（）(U+FF08, U+FF09)
   - 英文半角括号：() (U+0028, U+0029)

### 原因3：代码逻辑问题

代码已经被修复，现在使用更宽松的匹配逻辑：

```java
// 使用 contains 方式匹配，更容易匹配成功
if (trimmedValue.contains("一次性包装破损") && trimmedValue.contains("不支持")) {
    return "1300030902";
}
```

## 技术细节

### 导入流程

1. **读取Excel** → 从"无理由退货"列读取值
2. **解析标签** → `parseNoReasonReturnTagId()` 方法匹配标签ID
3. **存储到数据库** → 保存到 `no_reason_return_tag_id` 字段
4. **同步布尔值** → 如果是"1300030901"，设置 `no_reason_return=1`，否则为 0

### 导出流程

1. **从数据库读取** → 优先读取 `no_reason_return_tag_id`
2. **如果为空** → 根据 `no_reason_return` 映射：
   - 1 → "1300030901"
   - 0 → "1300030895"
3. **写入Excel** → 写入"无理由退货"列

## 预防措施
